Требования к разработке модуля

  1. Единая нотация:
    • Все названия методов, переменных и комментариев должны быть на русском языке, чтобы соответствовать текущему стилю проекта.
    • Смешивание русского и английского (например, в названиях методов) недопустимо, пока проект полностью не перейдёт на международный формат.
  2. Чистые геттеры:
    • Для методов, которые только возвращают данные (геттеры), использовать простые названия без префикса "Получить", например, Статус() вместо ПолучитьСтатус(). Это делает код более лаконичным и естественным для русской нотации.
  3. Метод Статус() как основа:
    • Каждый модуль должен начинаться с метода Статус(), который возвращает структуру с двумя основными полями:
      • Переменные — состояние всех переменных объекта.
      • Методы — список всех методов модуля с их описаниями.
    • Этот метод заменяет отдельные методы для отладки (например, ПолучитьСостояниеДляОтладки() или ПолучитьСписокМетодов()), чтобы избежать дублирования функциональности.
    • Метод Статус() размещается на верхнем уровне в #Область ПрограммныйИнтерфейс, так как он является универсальной точкой доступа для отладки.
  4. Шапка модуля:
    • Шапка должна содержать:
      • Название модуля и его тип (например, "МодульОбъектаОбработки").
      • Краткое описание функциональности модуля.
      • Автор, дата создания, версия, дата версии и зависимости (если есть).
      • Список всех доступных методов с кратким описанием (без параметров, чтобы не перегружать шапку).
    • Поле "Дата версии" должно обновляться при каждом изменении версии модуля, чтобы отражать дату последнего изменения версии.
    • Пример:
      text
      // МодульОбъектаОбработки: <Название>
      // Описание: <Краткое описание>
      // Автор: <Имя>
      // Дата создания: <Дата>
      // Версия: <Версия>
      // Дата версии: <Дата последнего изменения версии>
      // Зависимости: <Список или "НЕТ">
      //
      // Доступные методы:
      // - Статус() - Возвращает состояние объекта и список методов.
      // - <Другие методы с описанием>.
  5. Структура модуля:
    • Использовать стандартные области 1С:
      • #Область ОписаниеПеременных — для объявления всех переменных модуля с комментариями.
      • #Область ПрограммныйИнтерфейс — для публичных методов.
        • #Область Реквизиты — подчинённая область для геттеров и сеттеров, отвечающих за управление состоянием объекта (например, установка/получение значений переменных).
        • #Область Методы — подчинённая область для методов действия, реализующих логику поведения объекта.
      • #Область СлужебныеПроцедурыИФункции — для внутренних (приватных) методов.
      • #Область Тесты — для тестов (если модуль требует тестирования).
      • #Область Примеры — для примеров использования.
      • #Область Инициализация — для инициализации переменных.
    • Все методы в #Область ПрограммныйИнтерфейс должны быть помечены как Экспорт и иметь подробные комментарии с описанием параметров и возвращаемого значения.
  6. Поддержка цепочки вызовов:
    • Методы, изменяющие состояние объекта (сеттеры), должны возвращать ЭтотОбъект, чтобы поддерживать цепочку вызовов (например, Объект.Метод1().Метод2()).
  7. Тестирование и примеры:
    • Модуль должен содержать тесты для проверки ключевой функциональности.
    • Должен быть пример использования, демонстрирующий основные сценарии работы с модулем.
  8. Оптимизация кода:
    • Избегать дублирования функциональности. Если метод может быть покрыт другим (например, Статус() покрывает отладочные методы), избыточные методы нужно удалять.
    • Код должен быть максимально лаконичным, но читаемым.
